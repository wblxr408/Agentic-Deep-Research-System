# Agent 幻觉治理面试拷打笔记

> 版本：v1.0  
> 主题：把原有研究闭环，升级成“可控、可审、可拒答”的 Agent 护栏系统

## 1. 一句话总览

原方案解决的是“怎么让 Agent 跑起来并自校验”，你的思路补的是“怎么让 Agent 不乱说、不乱调、不乱做”。
这些护栏案例只是说明手段，不改变本项目最初的主目标：围绕“搜索、检索、对比、分析”做研究型 Agent。

## 2. 原方案怎么做

### 2.1 核心链路

- `Planner` 先把任务拆成 DAG。
- `Search / Browser / RAG` 负责取证。
- `Analyst` 基于证据生成分析。
- `Reflection` 对分析做事实、一致性、完整性校验。
- `Replan` 在校验失败时重规划。
- `Report` 输出最终报告。

### 2.2 原方案已经覆盖的幻觉控制

- 知识幻觉：靠检索证据 + 引用 + Reflection 校验。
- 工具幻觉：靠工具节点和调用历史，减少“凭空调用”。
- 证据幻觉：靠 `collected_evidence` 和 citation 链路约束输出。
- 行动幻觉：靠 DAG 路由和重规划循环约束行动。

### 2.3 原方案的局限

- 主要依赖流程和提示词。
- 没有统一的任务级 prompt 路由。
- 工具暴露面还不够细。
- 高风险动作的审批和回滚不是一等公民。
- 输出校验更多是 Reflection，缺少更硬的规则门禁。
- 但项目目标不变，核心仍然是搜索对比和证据驱动研究，这些护栏只是保证它不跑偏。

## 3. 你的增强思路

### 3.1 总原则

- 不知道就说不知道。
- 只能基于工具结果和检索文档回答。
- 文档没有提到的信息不要补充。
- 高风险动作必须先确认。
- 输出必须包含来源或置信度。

### 3.2 四层护栏

#### 输入层

- 先做任务分类和风险分级。
- 再从内置 prompt 模板里选最合适的一版。
- 不让模型“自由发挥角色设定”。

#### 工具层

- 工具白名单：当前任务只暴露当前需要的工具。
- 参数 schema 校验：类型、必填、枚举、格式都要限制。
- 高风险工具和只读工具分开。

#### 生成层

- 先检索，再回答。
- 检索不到就拒答，不硬编。
- 召回后先 Rerank，再把最相关证据喂给模型。
- 回答必须带来源或置信度。

#### 审查层

- 结构化输出校验。
- 答案与证据一致性校验。
- 敏感内容额外校验。
- 高风险场景加 Reviewer 或人工介入。

## 4. 五类幻觉怎么治

### 4.1 知识幻觉

问题：模型凭记忆补业务事实、公司规则、最新政策。

方案：
- 业务事实先检索。
- 只基于文档和工具结果回答。
- 找不到证据就拒答。
- 引用必须可追溯。

补充说明：
- 内部知识库优先于联网搜索。
- 内部知识库命中后，是否继续联网搜索由策略决定，不是默认跳过或默认强行继续。
- 内部知识源支持手工维护分组，避免所有文档混成一团。

### 4.2 工具幻觉

问题：模型假装调用了工具，或者编造工具结果。

方案：
- 工具白名单。
- 工具参数必须过 schema。
- Observation 只能来自真实工具返回。
- 不允许模型自己造“查询结果如下”。

### 4.3 参数幻觉

问题：模型把工具参数乱填、乱传、乱猜。

方案：
- 必填字段约束。
- 枚举值约束。
- 类型和格式约束。
- 参数不合法直接拦截，不进工具执行。

### 4.4 证据幻觉

问题：回答看似有理，但证据里根本没有。

方案：
- 检索结果先 Rerank。
- 只喂高相关证据。
- 输出后做答案-证据一致性校验。
- 数字、时间、限制条件逐条对齐。

### 4.5 行动幻觉

问题：模型想做写操作、删除、退款、发消息等高风险动作。

方案：
- 只读工具和写工具分离。
- 高风险工具必须先审批。
- 必要时支持回滚、幂等和操作日志。
- 没确认前，不能执行。

## 7. 面试边界

- 这个项目本质仍是研究型 Agent，不是通用执行型 Agent。
- 上面的写操作例子只是用来说明护栏设计，不改变项目最初的搜索对比目标。
- 现阶段已把核心护栏接入主链路，reviewer 节点还没独立成工作流分支，但确认、阻断、追踪和基础指标已经接入。

## 5. 面试追问答案

### Q1：为什么要做 prompt 路由？

因为不同任务的风险不同。FAQ、业务规则、财务事实、写操作，不能用同一套 prompt。路由的本质是把“能说什么、不能说什么”前置。

### Q2：为什么要工具白名单？

因为把所有工具都暴露给 Agent，会放大误调用和误操作。当前任务只给当前需要的工具，能显著降低动作幻觉和参数幻觉。

### Q3：为什么要 schema 校验？

因为工具调用不是聊天，是接口调用。类型、必填、枚举、格式不校验，Agent 很容易把“看起来合理”的参数传进去。

### Q4：为什么 Observation 不能由模型生成？

因为那样工具链就失真了。Observation 必须来自真实执行结果，否则模型可以把任何幻觉伪装成工具反馈。

### Q5：为什么检索不到就要拒答？

因为“没有证据”比“错误回答”更安全。业务规则、公司政策、最新数据都不能靠记忆补。

### Q6：为什么还要 Reflection？

因为前面的规则只能挡住一部分问题。Reflection 是最后一道审查，负责查漏补缺和触发重规划。

## 6. 面试总结句

原方案解决“Agent 怎么跑”，你的方案解决“Agent 怎么不乱跑、乱说、乱做”。两者合起来，才是可上线的 Agent 护栏系统。
